# Anonymous Event Check-In (Midnight / Compact)

Level 1 submission for the New Moon to Full challenge.

## What this is

A tiny Compact contract for checking people into an event without ever
recording *who* checked in. The public ledger only ever sees a running
counter and a one-way commitment — never a name, wallet, or secret code.

Think of a meetup, a conference, or a community event where the organizer
wants a verifiable, tamper-proof attendee count (for a sponsor report, a
capacity limit, whatever) but doesn't need or want a guest list sitting on
a public blockchain forever.

## The idea, longer version

Most "check-in" apps are really just attendance databases with extra
steps — a spreadsheet with a QR code bolted on. Even blockchain versions
of this usually end up putting a wallet address on-chain per attendee,
which quietly turns "I went to this event" into a permanent, linkable,
public record.

This contract flips that: each attendee holds a private secret code that
never leaves their device. Checking in proves "I have a valid, unused
code" and bumps a public counter — but the only thing that ever touches
the chain is a hash built from that code plus the current slot number, so
the same person checking into two different events (or the same event
twice, if that were allowed) produces two completely unlinkable
commitments. Nobody watching the chain — not even the organizer — can
tell who showed up, only how many people did.

Where I'd take this past Level 1: an organizer-only key for `closeEvent`,
a Merkle allow-list so only pre-registered codes can check in, and a small
front end that generates + QR-encodes the secret code for each attendee.

## Public ledger state vs. private witness

This is the core privacy mechanism in Compact, and it's the whole point
of this contract, so it's worth spelling out plainly:

**Public ledger state** (`eventState`, `attendeeCount`, `lastCheckIn`,
`capacity`) lives on-chain. Anyone running a Midnight node can read it.
It's what makes the contract trustworthy — nobody has to take the
organizer's word for the attendee count.

**Private witness** (`secretCode`) is a function the *user's own machine*
answers, not the contract. Its return value is private by default —
Compact won't let you write it to the ledger, return it from a circuit,
or hand it to another contract without wrapping it in `disclose()` first.
That's a compiler-enforced rule, not a convention: try to leak a witness
value by accident and `compact compile` throws
`potential witness-value disclosure must be declared but is not`.

In `checkIn()`, the secret code is only ever used *inside* the zero-knowledge
proof, to compute `attendeeCommitment(secretCode(), slot)`. That commitment
(not the code itself) is what gets wrapped in `disclose()` and written to
`lastCheckIn`. So the chain gets proof that a valid, correctly-computed
commitment was produced — without ever seeing the input that produced it.

## Setup — how to run this locally

### 1. Prerequisites

- [Install the Compact toolchain](https://docs.midnight.network/getting-started/installation)
  (this gets you the `compact` compiler and the Compact VS Code extension)
- A [proof server](https://docs.midnight.network/getting-started/installation#set-up-the-proof-server)
  running locally (needed to actually generate proofs, not strictly needed
  just to compile)
- Node.js 22+ — check with `node --version`
- Docker (used by the proof server)

### 2. Install dependencies

```bash
cd contract
npm install
```

### 3. Compile the contract

```bash
npm run compact
```

This runs `compact compile src/checkin.compact ./src/managed/checkin` and
should print something like:

```
Compiling 3 circuits:
  circuit "checkIn" (k=..., rows=...)
  circuit "closeEvent" (k=..., rows=...)
  circuit "attendeeCommitment" (k=..., rows=...)
Overall progress [====================] 3/3
```

Take your **compile screenshot** here — this is the "successful compile
output (circuits listed)" screenshot the checklist wants.

This also generates `src/managed/checkin/`, containing the compiled
circuits, the ZK keys, and the generated TypeScript API — this is the
"managed/ directory present (circuits + keys)" requirement.

### 4. Run the tests

```bash
npm run build
npm test
```

### 5. Deploy to Preview / Preprod

Follow the [deployment guide](https://docs.midnight.network/getting-started/installation)
to point your DApp / CLI config at Preprod, deploy the contract, and note
the resulting contract address. Take your **deployment screenshot** here
— address visible — for the "contract deployed with address shown" item.
